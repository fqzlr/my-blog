# Java 异常：从入门到企业级实践

> Java 异常机制是程序健壮性的核心防线。本文从异常体系分类出发，讲清楚 checked 与 unchecked 的边界、try-with-resources 的底层原理、自定义异常的设计原则，最终落地到企业级项目中的异常规范与统一处理方案。

## 引子

刚写 Java 的时候，编译器报错"unhandled exception"，于是加上 `try-catch`，里面写一句 `e.printStackTrace()`，程序跑起来了，觉得异常处理就这么简单。直到线上出现一条 NPE 日志——只有堆栈，没有业务上下文，排查 3 小时才定位到是某个字段为 null 导致的空指针。

这不是个例。异常处理是初学者最早接触但最晚真正理解的语言特性之一。多数人停留在"能编译通过"的阶段，而企业级开发对异常的要求是：**可追溯、可恢复、可监控**。

## 背景知识

### 异常体系的继承结构

Java 所有异常的根类是 `java.lang.Throwable`，它有两个直接子类：

- **Error**：JVM 层面的严重问题，程序不应捕获。例如 `OutOfMemoryError`、`StackOverflowError`
- **Exception**：程序层面的问题，可以被合理处理

`Exception` 再分为两类：

- **Checked Exception**（受检异常）：继承自 `Exception` 但不继承 `RuntimeException`，编译器强制要求处理
- **Unchecked Exception**（非受检异常）：继承自 `RuntimeException`，编译器不强制处理

[配图：Java 异常体系继承结构图，展示 Throwable → Error / Exception → RuntimeException 的层级关系，标注 checked 与 unchecked 的分界线]

### Checked vs Unchecked：设计意图与争议

Checked Exception 的设计意图是让调用者显式处理可预见的异常情况，编译器充当安全网。典型代表：`IOException`、`SQLException`。

但在实际项目中，Checked Exception 带来了两个问题：

1. **异常声明的传染性**：底层方法抛出 checked exception，中间每一层调用都要声明或捕获，导致方法签名膨胀
2. **过度捕获**：开发者为了通过编译，用空的 `catch` 块吞掉异常，比不处理更危险

Bruce Eckel（《Thinking in Java》作者）和 Rod Johnson（Spring 创始人）都公开批评过 Checked Exception 的设计。Spring 的数据访问模块将所有 `SQLException` 统一转为 `DataAccessException`（unchecked），正是基于这一判断。

**判断原则**：如果调用者能从异常中恢复，用 checked；如果恢复不了或调用者无法合理处理，用 unchecked。企业级项目中，**优先使用 unchecked exception** 已成为主流实践。

## 核心内容

### 常见运行时异常

| 异常类型 | 触发场景 | 典型代码 |
|----------|----------|----------|
| `NullPointerException` | 对 null 引用调用方法或访问字段 | `String s = null; s.length()` |
| `ClassCastException` | 强制转换到不兼容的类型 | `(String) new Object()` |
| `IllegalArgumentException` | 方法接收到不合法的参数值 | `Thread.sleep(-1)` |
| `ArithmeticException` | 非法算术运算（整数除零） | `int x = 1 / 0` |
| `IndexOutOfBoundsException` | 数组或集合下标越界 | `list.get(list.size())` |
| `NumberFormatException` | 字符串无法解析为数字 | `Integer.parseInt("abc")` |
| `UnsupportedOperationException` | 调用不支持的操作 | `Collections.unmodifiableList(list).add(1)` |
| `ArrayStoreException` | 向数组存入类型不兼容的元素 | `Object[] arr = new String[1]; arr[0] = 1` |
| `NegativeArraySizeException` | 创建负长度的数组 | `new int[-1]` |
| `SecurityException` | 安全管理器拒绝操作 | 沙箱环境中访问文件系统 |

这些异常的共同特征：继承自 `RuntimeException`，编译器不强制捕获。它们反映的是**程序逻辑错误**，而非外部环境问题，正确的做法是修复代码而非捕获异常。

### try-catch-finally 的执行顺序

执行顺序有 3 条规则，面试和实际开发中都会遇到：

1. `try` 中发生异常后，异常之后的代码不执行，跳转到对应的 `catch`
2. `finally` 总是执行（即使 `try` 或 `catch` 中有 `return`）
3. `finally` 中的 `return` 会覆盖 `try`/`catch` 中的 `return` 值

```java
public static int testReturn() {
    int i = 1;
    try {
        return i;
    } finally {
        i = 2;
    }
}
```

这段代码返回 **1**，不是 2。原因是 `return i` 执行时，返回值已经保存到局部变量表的专用槽位，`finally` 中修改 `i` 不影响已保存的返回值。但如果 `finally` 中写 `return i`，则返回 2，且会吞掉 `try` 块中的异常。

**规则：永远不要在 `finally` 中使用 `return`。**

### try-with-resources 与 AutoCloseable

JDK 7 引入的语法糖，解决 `finally` 中关闭资源的样板代码问题：

```java
try (FileInputStream fis = new FileInputStream("data.txt");
     BufferedReader br = new BufferedReader(new InputStreamReader(fis))) {
    return br.readLine();
}
```

编译后等价于：

```java
FileInputStream fis = null;
BufferedReader br = null;
try {
    fis = new FileInputStream("data.txt");
    br = new BufferedReader(new InputStreamReader(fis));
    return br.readLine();
} finally {
    if (br != null) br.close();
    if (fis != null) fis.close();
}
```

底层原理：所有实现了 `AutoCloseable` 接口的类都可以用在 `try-with-resources` 中。`AutoCloseable` 是 `Closeable` 的父接口，唯一的 `close()` 方法允许抛出 `Exception`。

如果 `try` 块和 `close()` 都抛出异常，`close()` 的异常会被**抑制**（suppressed），通过 `Throwable.getSuppressed()` 获取。这是 `try-with-resources` 相比手写 `finally` 的一个优势——不会丢失异常信息。

### throw 与 throws 的区别

| 关键字 | 作用位置 | 功能 |
|--------|----------|------|
| `throw` | 方法体内部 | 主动抛出一个异常实例 |
| `throws` | 方法签名声明 | 声明该方法可能抛出的异常类型 |

```java
public void setAge(int age) throws IllegalArgumentException {
    if (age < 0 || age > 150) {
        throw new IllegalArgumentException("年龄不合法: " + age);
    }
    this.age = age;
}
```

`throws` 对 unchecked exception 可写可不写（编译器不强制），但写上去是文档——告诉调用者这个方法可能抛出什么。

### 自定义异常的设计原则

企业级项目中，JDK 内置异常不足以表达业务语义。自定义异常需要遵循以下原则：

**原则 1：按业务领域划分异常层级**

```java
public class BizException extends RuntimeException {
    private final String errorCode;

    public BizException(String errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    public BizException(String errorCode, String message, Throwable cause) {
        super(message, cause);
        this.errorCode = errorCode;
    }

    public String getErrorCode() {
        return errorCode;
    }
}

public class OrderException extends BizException {
    public OrderException(String errorCode, String message) {
        super(errorCode, message);
    }
}

public class PaymentException extends BizException {
    public PaymentException(String errorCode, String message) {
        super(errorCode, message);
    }
}
```

**原则 2：继承 `RuntimeException`，不继承 `Exception`**

企业级项目通常在统一拦截层处理异常（Spring 的 `@ControllerAdvice`），中间层不需要逐层声明。Checked Exception 在 Web 应用中几乎没有恢复场景。

**原则 3：必须保留原始异常链**

```java
try {
    orderService.create(request);
} catch (RemoteException e) {
    throw new OrderException("ORDER_CREATE_FAILED", "订单创建失败", e);
}
```

第三个参数 `e` 不能省略。省略后，排查时看不到远程调用的原始堆栈，定位问题只能靠猜。

**原则 4：错误码与异常类分离**

不要为每种错误创建一个异常类（`OrderNotFoundException`、`OrderExpiredException`……），类会爆炸。用错误码区分具体场景，异常类按业务领域聚合：

```java
public class OrderErrorCode {
    public static final String NOT_FOUND = "ORDER_001";
    public static final String EXPIRED = "ORDER_002";
    public static final String STOCK_INSUFFICIENT = "ORDER_003";
}
```

## 实战演示

### 企业级统一异常处理（Spring Boot）

以下是一个完整的异常处理方案，覆盖 Web 层拦截、错误码管理、日志记录：

**1. 定义业务异常基类**

```java
@Getter
public class BizException extends RuntimeException {
    private final String errorCode;

    public BizException(String errorCode, String message) {
        super(message);
        this.errorCode = errorCode;
    }

    public BizException(String errorCode, String message, Throwable cause) {
        super(message, cause);
        this.errorCode = errorCode;
    }
}
```

**2. 定义全局错误码**

```java
public interface ErrorCode {
    String getCode();
    String getMessage();
}

public enum CommonErrorCode implements ErrorCode {
    PARAM_INVALID("COMMON_001", "参数校验失败"),
    UNAUTHORIZED("COMMON_002", "未授权"),
    FORBIDDEN("COMMON_003", "无访问权限"),
    INTERNAL_ERROR("COMMON_999", "系统内部错误");

    private final String code;
    private final String message;

    CommonErrorCode(String code, String message) {
        this.code = code;
        this.message = message;
    }

    @Override
    public String getCode() { return code; }

    @Override
    public String getMessage() { return message; }
}
```

**3. 统一响应体**

```java
@Data
@AllArgsConstructor
@NoArgsConstructor
public class ApiResponse<T> {
    private String code;
    private String message;
    private T data;

    public static <T> ApiResponse<T> success(T data) {
        return new ApiResponse<>("0", "success", data);
    }

    public static <T> ApiResponse<T> fail(String code, String message) {
        return new ApiResponse<>(code, message, null);
    }
}
```

**4. 全局异常处理器**

```java
@RestControllerAdvice
@Slf4j
public class GlobalExceptionHandler {

    @ExceptionHandler(BizException.class)
    public ApiResponse<Void> handleBizException(BizException e) {
        log.warn("业务异常: code={}, msg={}", e.getErrorCode(), e.getMessage());
        return ApiResponse.fail(e.getErrorCode(), e.getMessage());
    }

    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ApiResponse<Void> handleValidation(MethodArgumentNotValidException e) {
        String msg = e.getBindingResult().getFieldErrors().stream()
            .map(fe -> fe.getField() + ": " + fe.getDefaultMessage())
            .collect(Collectors.joining("; "));
        return ApiResponse.fail(CommonErrorCode.PARAM_INVALID.getCode(), msg);
    }

    @ExceptionHandler(Exception.class)
    public ApiResponse<Void> handleUnknown(Exception e) {
        log.error("未处理异常", e);
        return ApiResponse.fail(CommonErrorCode.INTERNAL_ERROR.getCode(),
            CommonErrorCode.INTERNAL_ERROR.getMessage());
    }
}
```

**5. 业务层使用**

```java
@Service
public class OrderService {

    public Order create(CreateOrderRequest req) {
        Product product = productRepository.findById(req.getProductId())
            .orElseThrow(() -> new BizException("ORDER_001", "商品不存在"));
        if (product.getStock() < req.getQuantity()) {
            throw new BizException("ORDER_003", "库存不足");
        }
        Order order = Order.create(product, req.getQuantity());
        return orderRepository.save(order);
    }
}
```

整个链路：业务层抛出 `BizException` → `GlobalExceptionHandler` 捕获 → 返回统一格式的错误响应。前端根据 `code` 字段做对应处理，日志中保留完整堆栈用于排查。

## 踩坑点 & 注意事项

### 吞掉异常

```java
try {
    riskyOperation();
} catch (Exception e) {
    // 什么都不做
}
```

这是最危险的做法。异常被静默吞掉，上层调用者认为操作成功，数据不一致就在这种地方埋下。至少要记录日志：

```java
} catch (Exception e) {
    log.error("操作失败, param={}", param, e);
    throw new BizException("OP_FAILED", "操作失败", e);
}
```

### 异常信息泄露

```java
@ExceptionHandler(Exception.class)
public ApiResponse<Void> handle(Exception e) {
    return ApiResponse.fail("999", e.getMessage());
}
```

`e.getMessage()` 可能包含 SQL 语句、文件路径、内部 IP 等敏感信息。对未知异常，统一返回"系统内部错误"，详细信息只写日志。

### 在循环中频繁创建异常对象

异常对象的 `fillInStackTrace()` 调用是同步的，且会遍历整个调用栈，开销不小。在每秒百万次调用的热路径中，用异常做流程控制（如用 `NumberFormatException` 判断字符串是否为数字）会带来显著的性能损耗。

**替代方案**：用返回值做判断，异常只用于真正的异常场景。

```java
// 错误：用异常做流程控制
try {
    Integer.parseInt(input);
    return true;
} catch (NumberFormatException e) {
    return false;
}

// 正确：用工具方法
return NumberUtils.isParsable(input);
```

### finally 中的异常覆盖 try 中的异常

```java
try {
    throw new SQLException("数据库连接失败");
} finally {
    throw new IOException("关闭连接失败");
}
```

调用者只能捕获到 `IOException`，`SQLException` 被丢失。`try-with-resources` 通过 suppressed 机制解决了这个问题——这也是优先使用 `try-with-resources` 的理由之一。

### 多 catch 块的顺序

JDK 7 支持 multi-catch 语法：`catch (IOException | SQLException e)`。但 `catch` 块的顺序仍然有约束：子类异常必须在父类异常之前，否则编译报错。

```java
// 编译错误：SQLException 是 Exception 的子类，不能放在后面
catch (Exception e) { ... }
catch (SQLException e) { ... }

// 正确顺序
catch (SQLException e) { ... }
catch (Exception e) { ... }
```

## 总结

1. Java 异常体系分 Error / Checked Exception / Unchecked Exception 三层，企业级项目**优先使用 Unchecked Exception**
2. `try-with-resources` 比 `try-finally` 更安全（不会丢失 suppressed 异常）也更简洁，只要资源实现了 `AutoCloseable` 就应该用它
3. 自定义异常按业务领域划分层级，错误码与异常类分离，避免类爆炸
4. 异常链必须保留原始 cause，`finally` 中不能 `return`，不能吞掉异常——这三条违反任何一条都会导致排查困难
5. 统一异常处理（`@ControllerAdvice` + 错误码枚举）是企业级项目的标配，前端根据错误码做业务逻辑，后端日志保留完整堆栈
