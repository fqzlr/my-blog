# MongoDB

> 从关系型数据库转向 MongoDB，核心转变是"先有数据，再定结构"。本文从 NoSQL 的定位出发，逐步覆盖 MongoDB 的文档模型、查询语法、聚合管道、索引策略、Java 集成，最终落到企业级的副本集与分片方案。

[配图：NoSQL数据库分类全景图——展示键值、文档、列族、图四种NoSQL类型的代表产品与数据模型对比]

## 为什么需要 NoSQL

关系型数据库（MySQL、PostgreSQL）在事务一致性和结构化查询上表现成熟，但面对以下场景时存在瓶颈：

| 场景 | 关系型数据库的局限 | NoSQL 的应对 |
|------|-------------------|-------------|
| 数据结构频繁变动 | DDL 变更需锁表，影响线上服务 | Schema-less，字段增减无需迁移 |
| 海量半结构化数据 | 行存储对 JSON/嵌套结构需拆表 | 文档模型天然支持嵌套 |
| 水平扩展需求 | 分库分表方案复杂，中间件成本高 | 原生支持分片，自动数据分布 |
| 高吞吐写入 | 单机写入上限受磁盘和锁制约 | 分片后写入分散到多节点 |

NoSQL 不是替代关系型数据库，而是在特定场景下的互补选择。MongoDB 作为文档型 NoSQL 的代表，在 Java 企业开发中使用广泛。

## 核心概念

### 文档（Document）

MongoDB 的基本数据单元是文档，格式为 BSON（Binary JSON）。BSON 在 JSON 基础上增加了日期、二进制、ObjectId 等类型。

```json
{
  "_id": ObjectId("6507f1e8c9a1b2d3e4f5a6b7"),
  "name": "张三",
  "age": 28,
  "address": {
    "city": "北京",
    "district": "海淀"
  },
  "tags": ["Java", "MongoDB"],
  "createdAt": ISODate("2026-01-12T08:00:00Z")
}
```

关键点：
- 每个文档有唯一的 `_id` 字段，未指定时自动生成 12 字节 ObjectId
- 文档支持嵌套（如上例的 `address`），无需像关系型数据库那样拆表
- 单个文档最大 16MB（MongoDB 6.x 默认限制）

### 集合（Collection）

集合是文档的容器，对应关系型数据库的"表"。集合无固定 Schema，同一集合内的文档结构可以不同。

### 数据库（Database）

数据库是集合的容器。MongoDB 实例可包含多个数据库，每个数据库有独立的文件集合和权限。

[配图：MongoDB数据层级结构——Database → Collection → Document → Field的嵌套关系图]

### 与关系型数据库的术语对照

| 关系型数据库 | MongoDB | 说明 |
|-------------|---------|------|
| Database | Database | 概念一致 |
| Table | Collection | 表 vs 集合 |
| Row | Document | 行 vs 文档 |
| Column | Field | 列 vs 字段 |
| Primary Key | `_id` | 主键 vs ObjectId |
| JOIN | `$lookup` | 聚合管道中的关联操作 |
| Index | Index | 概念一致 |

## CRUD 操作

### 插入（Create）

```javascript
// 插入单条
db.users.insertOne({
  name: "张三",
  age: 28,
  address: { city: "北京", district: "海淀" },
  tags: ["Java", "MongoDB"]
});

// 批量插入
db.users.insertMany([
  { name: "李四", age: 25, tags: ["Python"] },
  { name: "王五", age: 30, tags: ["Go", "Docker"] }
]);
```

`insertMany` 支持有序写入（默认，遇错停止）和无序写入（`ordered: false`，遇错跳过继续），大批量写入时无序模式吞吐更高。

### 查询（Read）

```javascript
// 等值查询
db.users.find({ name: "张三" });

// 嵌套字段查询
db.users.find({ "address.city": "北京" });

// 数组元素查询（tags 包含 "Java" 的文档）
db.users.find({ tags: "Java" });

// 比较操作符
db.users.find({ age: { $gte: 25, $lte: 30 } });

// 逻辑操作符
db.users.find({
  $and: [
    { "address.city": "北京" },
    { age: { $gte: 25 } }
  ]
});

// 投影（只返回 name 和 age 字段，排除 _id）
db.users.find({ age: { $gte: 25 } }, { name: 1, age: 1, _id: 0 });
```

常用查询操作符：

| 操作符 | 含义 | 示例 |
|--------|------|------|
| `$eq` | 等于 | `{ age: { $eq: 28 } }` |
| `$ne` | 不等于 | `{ age: { $ne: 28 } }` |
| `$gt` / `$gte` | 大于 / 大于等于 | `{ age: { $gt: 25 } }` |
| `$lt` / `$lte` | 小于 / 小于等于 | `{ age: { $lte: 30 } }` |
| `$in` | 在列表中 | `{ age: { $in: [25, 28, 30] } }` |
| `$nin` | 不在列表中 | `{ age: { $nin: [25, 28] } }` |
| `$regex` | 正则匹配 | `{ name: { $regex: /^张/ } }` |
| `$exists` | 字段是否存在 | `{ address: { $exists: true } }` |

### 更新（Update）

```javascript
// 更新单条（$set 修改指定字段，其余字段不变）
db.users.updateOne(
  { name: "张三" },
  { $set: { age: 29, "address.district": "朝阳" } }
);

// 更新多条
db.users.updateMany(
  { "address.city": "北京" },
  { $inc: { age: 1 } }
);

// 替换整条文档（除 _id 外全部替换）
db.users.replaceOne(
  { name: "张三" },
  { name: "张三", age: 29, email: "zhangsan@example.com" }
);
```

常用更新操作符：

| 操作符 | 含义 | 示例 |
|--------|------|------|
| `$set` | 设置字段值 | `{ $set: { age: 29 } }` |
| `$unset` | 删除字段 | `{ $unset: { email: "" } }` |
| `$inc` | 数值增减 | `{ $inc: { age: 1 } }` |
| `$push` | 数组追加元素 | `{ $push: { tags: "Redis" } }` |
| `$pull` | 数组移除元素 | `{ $pull: { tags: "Java" } }` |
| `$addToSet` | 数组去重追加 | `{ $addToSet: { tags: "Go" } }` |

### 删除（Delete）

```javascript
// 删除单条
db.users.deleteOne({ name: "王五" });

// 删除多条
db.users.deleteMany({ age: { $lt: 20 } });

// 清空集合（保留索引，速度比 drop 快）
db.users.deleteMany({});
```

## 聚合管道（Aggregation Pipeline）

聚合管道是 MongoDB 的核心查询能力，通过多阶段管道对文档进行逐级变换和计算。在报表统计、数据分析场景中替代应用层的数据处理逻辑。

[配图：聚合管道数据流转图——展示$match → $group → $sort → $limit各阶段对文档流的逐步过滤与变换]

### 常用阶段

| 阶段 | 作用 | 类比 SQL |
|------|------|---------|
| `$match` | 过滤文档 | WHERE |
| `$group` | 分组聚合 | GROUP BY |
| `$sort` | 排序 | ORDER BY |
| `$limit` | 限制条数 | LIMIT |
| `$project` | 字段投影 | SELECT |
| `$unwind` | 展开数组 | — |
| `$lookup` | 关联查询 | LEFT JOIN |
| `$count` | 计数 | COUNT(*) |

### 示例：统计各城市用户数量和平均年龄

```javascript
db.users.aggregate([
  // 阶段1：过滤有效用户
  { $match: { age: { $gte: 18 } } },

  // 阶段2：按城市分组，计算用户数和平均年龄
  {
    $group: {
      _id: "$address.city",
      userCount: { $sum: 1 },
      avgAge: { $avg: "$age" }
    }
  },

  // 阶段3：按用户数降序排列
  { $sort: { userCount: -1 } },

  // 阶段4：取前10
  { $limit: 10 }
]);
```

### 示例：$lookup 关联查询

```javascript
db.orders.aggregate([
  {
    $lookup: {
      from: "users",           // 关联的目标集合
      localField: "userId",    // 当前集合的关联字段
      foreignField: "_id",     // 目标集合的关联字段
      as: "userInfo"           // 输出字段名
    }
  },
  // userInfo 是数组，$unwind 展开为对象
  { $unwind: "$userInfo" },
  {
    $project: {
      orderId: 1,
      amount: 1,
      userName: "$userInfo.name"
    }
  }
]);
```

`$lookup` 的性能代价高于关系型数据库的 JOIN——它本质上是左表每条文档对右表做一次查询。数据量大时，应通过冗余字段（嵌入文档）减少 `$lookup` 使用。

## 索引

索引是 MongoDB 查询性能的关键。没有索引的查询执行全集合扫描（COLLSCAN），数据量超过 10 万条时性能急剧下降。

### 索引类型

| 类型 | 创建语法 | 适用场景 |
|------|---------|---------|
| 单字段索引 | `createIndex({ name: 1 })` | 单字段等值/范围查询 |
| 复合索引 | `createIndex({ city: 1, age: -1 })` | 多字段组合查询 |
| 多键索引 | 自动创建（索引字段为数组时） | 数组元素查询 |
| 文本索引 | `createIndex({ content: "text" })` | 全文检索 |
| 地理空间索引 | `createIndex({ location: "2dsphere" })` | LBS 附近查询 |
| 哈希索引 | `createIndex({ userId: "hashed" })` | 分片键、等值查询 |
| TTL 索引 | `createIndex({ createdAt: 1 }, { expireAfterSeconds: 3600 })` | 自动过期删除 |

### 复合索引与最左前缀

复合索引遵循最左前缀原则，与 MySQL 一致：

```javascript
// 创建复合索引
db.users.createIndex({ city: 1, age: -1, name: 1 });

// 命中索引的查询
db.users.find({ city: "北京" });                              // 命中 city
db.users.find({ city: "北京", age: { $gte: 25 } });          // 命中 city + age
db.users.find({ city: "北京", age: 25, name: "张三" });       // 命中全部三个字段

// 不命中索引的查询
db.users.find({ age: 25 });                                   // 跳过 city，无法命中
db.users.find({ name: "张三" });                              // 跳过 city 和 age，无法命中
```

### 索引使用建议

1. **ESR 原则**：复合索引字段顺序按 等值查询（Equality）→ 排序（Sort）→ 范围查询（Range）排列
2. **覆盖查询**：查询字段和投影字段都在索引中时，无需回表读取文档
3. **避免过度索引**：每个索引占用磁盘空间，写入时需同步更新所有索引。写入密集型集合的索引数量控制在 5 个以内
4. **定期审查**：使用 `$indexStats` 查看索引使用情况，删除无用索引

```javascript
// 查看查询执行计划
db.users.find({ city: "北京", age: { $gte: 25 } }).explain("executionStats");

// 关键指标：
// - winningPlan.stage = "IXSCAN" 表示命中索引
// - winningPlan.stage = "COLLSCAN" 表示全表扫描，需加索引
// - executionStats.totalDocsExamined 应接近返回条数
```

## Spring Data MongoDB 集成

### Maven 依赖

```xml
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-data-mongodb</artifactId>
</dependency>
```

### 配置

```yaml
spring:
  data:
    mongodb:
      uri: mongodb://username:password@localhost:27017/mydb?authSource=admin
      # 副本集连接串示例：
      # uri: mongodb://user:pass@node1:27017,node2:27017,node3:27017/mydb?replicaSet=rs0
```

### 实体定义

```java
import org.springframework.data.annotation.Id;
import org.springframework.data.mongodb.core.index.CompoundIndex;
import org.springframework.data.mongodb.core.index.Indexed;
import org.springframework.data.mongodb.core.mapping.Document;
import org.springframework.data.mongodb.core.mapping.Field;
import java.time.LocalDateTime;
import java.util.List;

@Document(collection = "users")
@CompoundIndex(name = "city_age_idx", def = "{'address.city': 1, 'age': -1}")
public class User {

    @Id
    private String id;

    @Indexed
    private String name;

    private Integer age;

    private Address address;

    private List<String> tags;

    @Field("created_at")
    private LocalDateTime createdAt;

    // getter/setter 省略
}

public class Address {
    private String city;
    private String district;
    // getter/setter 省略
}
```

`@Document` 指定集合名，`@CompoundIndex` 定义复合索引，`@Indexed` 定义单字段索引，`@Field` 映射字段名。

### Repository 方式

```java
public interface UserRepository extends MongoRepository<User, String> {

    List<User> findByAddressCity(String city);

    List<User> findByAgeBetween(Integer min, Integer max);

    List<User> findByTagsContaining(String tag);

    long countByAddressCity(String city);
}
```

Spring Data MongoDB 根据方法名自动派生查询。`findByAddressCity` 解析为 `{"address.city": city}` 的等值查询。

### MongoTemplate 方式

Repository 适合简单查询，复杂查询（聚合、动态条件、批量操作）使用 MongoTemplate：

```java
import org.springframework.data.mongodb.core.MongoTemplate;
import org.springframework.data.mongodb.core.aggregation.Aggregation;
import org.springframework.data.mongodb.core.query.Criteria;
import org.springframework.data.mongodb.core.query.Query;
import org.springframework.data.mongodb.core.query.Update;
import org.springframework.stereotype.Component;
import javax.annotation.Resource;
import java.util.List;

@Component
public class UserCustomRepository {

    @Resource
    private MongoTemplate mongoTemplate;

    public List<User> findUsers(String city, Integer minAge, Integer maxAge) {
        Query query = new Query();
        if (city != null) {
            query.addCriteria(Criteria.where("address.city").is(city));
        }
        if (minAge != null && maxAge != null) {
            query.addCriteria(Criteria.where("age").gte(minAge).lte(maxAge));
        }
        return mongoTemplate.find(query, User.class);
    }

    public void updateAge(String userId, Integer newAge) {
        Query query = new Query(Criteria.where("_id").is(userId));
        Update update = new Update().set("age", newAge);
        mongoTemplate.updateFirst(query, update, User.class);
    }

    public List<CityStats> groupByCity() {
        Aggregation aggregation = Aggregation.newAggregation(
            Aggregation.match(Criteria.where("age").gte(18)),
            Aggregation.group("address.city")
                .count().as("userCount")
                .avg("age").as("avgAge"),
            Aggregation.sort(org.springframework.data.domain.Sort.by(
                org.springframework.data.domain.Sort.Order.desc("userCount")
            )),
            Aggregation.limit(10)
        );
        return mongoTemplate.aggregate(aggregation, "users", CityStats.class).getMappedResults();
    }
}
```

### 批量写入

```java
import org.springframework.data.mongodb.core.BulkOperations;
import java.util.List;

@Component
public class UserBatchWriter {

    @Resource
    private MongoTemplate mongoTemplate;

    public void batchInsert(List<User> users) {
        BulkOperations bulkOps = mongoTemplate.bulkOps(
            BulkOperations.BulkMode.UNORDERED, User.class
        );
        for (User user : users) {
            bulkOps.insert(user);
        }
        bulkOps.execute();
    }
}
```

`BulkMode.UNORDERED` 对应 `ordered: false`，写入失败不影响其他文档。批量写入的吞吐量是逐条写入的 10-50 倍。

## 数据建模

MongoDB 的数据建模原则与关系型数据库不同，核心取舍是"嵌入 vs 引用"。

### 嵌入文档

```json
{
  "_id": ObjectId("..."),
  "name": "张三",
  "orders": [
    { "orderId": "ORD001", "amount": 99.9, "createdAt": ISODate("...") },
    { "orderId": "ORD002", "amount": 199.0, "createdAt": ISODate("...") }
  ]
}
```

适用条件：
- 子文档与父文档是强归属关系（一对一或一对少量）
- 子文档总大小不超过几 KB
- 查询时总是需要一起返回

限制：单个文档最大 16MB，嵌入数组无限增长会导致文档膨胀。

### 引用文档

```json
// users 集合
{ "_id": ObjectId("..."), "name": "张三" }

// orders 集合
{ "_id": ObjectId("..."), "userId": ObjectId("..."), "amount": 99.9 }
```

适用条件：
- 子文档数量多或增长不可控
- 子文档需要独立查询
- 子文档被多个父文档引用（多对多）

### 建模决策表

| 条件 | 嵌入 | 引用 |
|------|------|------|
| 子文档数量 | 少（< 100 条） | 多或不可控 |
| 子文档大小 | 小（< 1KB） | 大 |
| 读取模式 | 总是一起读 | 经常独立读 |
| 更新频率 | 低频整体更新 | 高频独立更新 |
| 归属关系 | 一对一 / 一对少量 | 一对多 / 多对多 |

## 企业级部署架构

### 副本集（Replica Set）

副本集是 MongoDB 高可用的基础，最少 3 个节点：

- **Primary**：接收所有写操作
- **Secondary**：从 Primary 同步数据（oplog），可配置为可读
- **Arbiter**：仲裁节点，不存数据，仅参与投票（节省资源）

[配图：MongoDB副本集架构——Primary节点写入、oplog同步到Secondary、自动故障转移流程]

写入流程：
1. 客户端写入 Primary
2. Primary 将操作记录到 oplog（固定大小集合）
3. Secondary 从 Primary 拉取 oplog 并重放
4. 根据写关注（Write Concern）决定何时返回成功

```javascript
// 写关注配置
db.users.insertOne(
  { name: "张三", age: 28 },
  { writeConcern: { w: "majority", j: true, wtimeout: 5000 } }
);
// w: "majority" — 等待多数节点确认
// j: true — 等待日志落盘（journal）
// wtimeout: 5000 — 超时 5 秒
```

读偏好（Read Preference）控制读请求路由：

| 模式 | 行为 | 场景 |
|------|------|------|
| `primary` | 只读 Primary（默认） | 强一致性要求 |
| `primaryPreferred` | 优先 Primary，不可用时读 Secondary | 一般场景 |
| `secondary` | 只读 Secondary | 报表查询，减轻 Primary 压力 |
| `secondaryPreferred` | 优先 Secondary | 读多写少场景 |
| `nearest` | 延迟最低的节点 | 对一致性要求低，对延迟敏感 |

### 分片集群（Sharded Cluster）

当单机或副本集无法承载数据量或吞吐量时，使用分片集群水平扩展。

[配图：MongoDB分片集群架构——mongos路由、Config Server、3个Shard（各为副本集）、数据按分片键分布]

核心组件：
- **mongos**：路由进程，接收客户端请求，根据分片键路由到对应 Shard
- **Config Server**：存储集群元数据（分片信息、chunk 分布），3 节点副本集
- **Shard**：数据分片，每个 Shard 是一个副本集

分片键选择是分片集群最关键的决策：

| 分片键类型 | 特点 | 适用场景 |
|-----------|------|---------|
| 范围分片 | 支持范围查询，但可能热点 | 有自然范围属性的字段（如日期） |
| 哈希分片 | 数据分布均匀，范围查询效率低 | 等值查询为主（如 userId） |
| 组合分片 | 兼顾分布和查询 | 常见组合：租户 ID + 时间 |

分片键选择原则：
1. **高基数**：取值范围大，避免数据倾斜（如不用布尔值）
2. **低频率**：单值出现频率低，避免热点 chunk
3. **非单调递增**：避免所有写入集中到最后一个 chunk（如 ObjectId 不适合做范围分片键）

### 连接池配置

```yaml
spring:
  data:
    mongodb:
      uri: mongodb://user:pass@node1:27017,node2:27017,node3:27017/mydb?replicaSet=rs0
      # 连接池参数（通过 URI 参数传递）
      # minPoolSize=10
      # maxPoolSize=100
      # maxWaitTime=5000
      # connectTimeoutMS=3000
      # socketTimeoutMS=0
```

| 参数 | 建议值 | 说明 |
|------|--------|------|
| `minPoolSize` | 10 | 最小连接数，避免冷启动延迟 |
| `maxPoolSize` | 50-100 | 最大连接数，按实例规格调整 |
| `maxWaitTime` | 5000ms | 连接池耗尽时的等待超时 |
| `connectTimeoutMS` | 3000ms | 建立连接超时 |
| `socketTimeoutMS` | 0（无限） | 读操作超时，0 表示由服务端控制 |

## 踩坑点 & 注意事项

### 文档大小超限

MongoDB 单文档上限 16MB。嵌入数组无限增长（如日志、评论）是常见触发原因。

解决方案：
- 大数组拆为独立集合，用引用关联
- 使用 Bucket Pattern：按时间或数量分组嵌入

```json
// Bucket Pattern：每小时一条文档，而非每条日志一条
{
  "_id": ObjectId("..."),
  "userId": "U001",
  "hour": ISODate("2026-01-12T08:00:00Z"),
  "logs": [
    { "time": ISODate("2026-01-12T08:01:12Z"), "action": "login" },
    { "time": ISODate("2026-01-12T08:15:33Z"), "action": "purchase" }
  ]
}
```

### 索引未命中的常见原因

1. **`$or` 查询**：`$or` 的每个条件需要独立索引，复合索引无法跨条件使用
2. **`$ne` / `$nin` / `$exists: false`**：这些操作符即使有索引也大概率走全表扫描
3. **类型不匹配**：字段存的是字符串 `"28"`，查询用数字 `28`，索引不命中
4. **前缀通配符正则**：`/^张/` 命中索引，`/张$/` 不命中

### 写关注与读一致性

默认写关注为 `w: 1`（Primary 确认即返回），此时立即从 Secondary 读可能读到旧数据（读己之写问题）。

解决方案：
- 写入后使用 `w: "majority"` + 读偏好 `primary`
- 或在会话中使用因果一致性（Causal Consistency）

### 大量 OR 查询性能差

```javascript
// 差：多个 $or 条件
db.orders.find({
  $or: [
    { status: "pending" },
    { status: "processing" },
    { status: "shipped" }
  ]
});

// 好：使用 $in
db.orders.find({ status: { $in: ["pending", "processing", "shipped"] } });
```

### Spring Data MongoDB 的 N+1 查询

使用 `@DBRef` 关联时，加载父文档后逐条加载引用文档，产生 N+1 查询。

解决方案：
- 优先使用嵌入文档或手动引用（存 ObjectId，用 `$lookup` 聚合查询）
- 避免 `@DBRef`，它在性能敏感场景下是反模式

## 性能对比

| 维度 | MongoDB | MySQL（InnoDB） |
|------|---------|-----------------|
| 写入吞吐（单节点） | 约 15,000 ops/s（简单文档） | 约 8,000 ops/s（简单行） |
| 读取延迟（索引命中） | < 1ms | < 1ms |
| 聚合查询 | 管道阶段串行执行，内存受限 | 优化器选择执行计划，成熟稳定 |
| 事务支持 | 4.0+ 支持多文档事务（性能损耗约 3-5 倍） | 原生 ACID，无额外损耗 |
| 水平扩展 | 原生分片，运维成本中等 | 需中间件（ShardingSphere 等），运维成本高 |
| Schema 变更 | 无需 DDL，应用层兼容即可 | 需 ALTER TABLE，大表可能锁表 |

数据来源：基于 AWS i3.xlarge 实例的基准测试，MongoDB 7.0 vs MySQL 8.0，2025-12 采样。

## 部署清单

生产环境 MongoDB 部署的必要配置项：

| 项目 | 配置 | 说明 |
|------|------|------|
| 副本集 | 至少 3 节点（1 Primary + 2 Secondary） | 高可用最低要求 |
| 认证 | `authorization: enabled` | 开启鉴权 |
| 网络加密 | TLS/SSL | 防止中间人攻击 |

| 备份 | `mongodump` 或云服务商快照 | 定期全量 + oplog 增量 |

## 总结

MongoDB 的核心价值在于文档模型对半结构化数据的天然适配，以及分片集群对水平扩展的原生支持。选型时需明确三点：

1. **数据是否适合文档模型**：嵌套结构、字段变动频繁 → 适合；强关联、多表 JOIN → 不适合
2. **事务需求**：单文档操作天然原子；跨文档事务可用但有性能损耗，高频事务场景优先选关系型数据库
3. **扩展路径**：数据量和写入量可预见地超过单机上限 → MongoDB 分片方案比 MySQL 分库分表运维成本低

Java 企业开发中，Spring Data MongoDB 提供了 Repository 和 MongoTemplate 两层抽象，简单 CRUD 用 Repository，复杂查询和聚合用 MongoTemplate，两者可混用。
