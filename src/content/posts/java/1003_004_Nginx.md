# Nginx 核心知识

> Nginx 是 Java 后端架构中不可或缺的基础设施层。本文从核心概念出发，覆盖反向代理、负载均衡、静态资源服务、HTTPS 配置和性能调优，配合 Spring Boot 实战演示。

## 引子

Java 后端服务通常运行在 8080 等非标准端口上，用户不会在浏览器输入 `example.com:8080` 访问。生产环境中，Nginx 充当流量入口，将 80/443 端口的请求转发到后端 Java 进程。除此之外，静态资源托管、SSL 终结、负载均衡、限流熔断都依赖 Nginx 完成。理解 Nginx 的配置和工作原理，是 Java 工程师部署和运维能力的基本要求。

## 核心概念

### 正向代理 vs 反向代理

[配图：正向代理与反向代理的请求流向对比图]

| 维度 | 正向代理 | 反向代理 |
|------|---------|---------|
| 代理对象 | 客户端 | 服务端 |
| 谁知道真实目标 | 客户端知道，代理帮客户端转发 | 客户端不知道，代理帮服务端接收 |
| 典型场景 | VPN、科学上网 | Nginx 转发到 Spring Boot |
| 客户端感知 | 知道代理存在 | 不知道代理存在 |

Nginx 在 Java 架构中扮演的角色是**反向代理**：客户端请求 `example.com`，Nginx 接收后将请求转发到 `127.0.0.1:8080` 的 Spring Boot 进程，客户端全程只与 Nginx 交互。

### Nginx 的核心能力

1. **反向代理**：接收客户端请求，转发到上游服务
2. **负载均衡**：将请求分发到多个上游实例
3. **静态资源服务**：直接返回 HTML/CSS/JS/图片，不走后端
4. **SSL 终结**：在 Nginx 层处理 HTTPS，后端只需 HTTP
5. **流量控制**：限流、熔断、请求排队

  --name nginx \

## 配置语法

### 配置文件结构

Nginx 配置采用嵌套块结构，从外到内依次为：

```
main        # 全局配置
└── events  # 连接处理配置
└── http    # HTTP 服务配置
    └── server    # 虚拟主机配置
        └── location  # 路由匹配配置
```

### 最小配置模板

```nginx
worker_processes auto;

events {
    worker_connections 1024;
}

http {
    include       mime.types;
    default_type  application/octet-stream;

    sendfile on;

    server {
        listen 80;
        server_name localhost;

        location / {
            root   html;
            index  index.html index.htm;
        }
    }
}
```

### location 匹配规则

[配图：location 匹配优先级的决策流程图]

| 修饰符 | 含义 | 优先级 | 示例 |
|--------|------|--------|------|
| `=` | 精确匹配 | 1（最高） | `location = /api/login` |
| `^~` | 前缀匹配，匹配后不检查正则 | 2 | `location ^~ /static/` |
| `~` | 正则匹配（区分大小写） | 3 | `location ~ \.php$` |
| `~*` | 正则匹配（不区分大小写） | 3 | `location ~* \.(jpg|png)$` |
| 无修饰符 | 前缀匹配 | 4（最低） | `location /api/` |

匹配优先级：精确匹配 `=` > 前缀匹配 `^~` > 正则匹配 `~`/`~*` > 普通前缀匹配。

```nginx
server {
    listen 80;

    # 精确匹配：只匹配 /api/login，不匹配 /api/login/
    location = /api/login {
        proxy_pass http://127.0.0.1:8080;
    }

    # 前缀匹配：匹配 /static/ 下所有路径，不检查正则
    location ^~ /static/ {
        alias /data/static/;
    }

    # 正则匹配：所有 .php 结尾的请求
    location ~ \.php$ {
        fastcgi_pass 127.0.0.1:9000;
    }

    # 普通前缀匹配：匹配 /api/ 下所有路径
    location /api/ {
        proxy_pass http://127.0.0.1:8080;
    }
}
```

## 反向代理

### 基础反向代理配置

将 Nginx 作为 Spring Boot 应用的反向代理：

```nginx
server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://127.0.0.1:8080;

        # 传递客户端真实信息
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```

`proxy_set_header` 是反向代理的关键配置。不加这些 Header，后端 Spring Boot 拿到的 IP 全部是 Nginx 的 `127.0.0.1`，无法做 IP 限流、地域统计。

### Spring Boot 端获取真实 IP

```java
import javax.servlet.http.HttpServletRequest;

@Component
public class RequestUtils {

    public String getClientIp(HttpServletRequest request) {
        String ip = request.getHeader("X-Forwarded-For");
        if (ip != null && !ip.isEmpty() && !"unknown".equalsIgnoreCase(ip)) {
            // X-Forwarded-For 可能有多个 IP，取第一个
            return ip.split(",")[0].trim();
        }
        ip = request.getHeader("X-Real-IP");
        if (ip != null && !ip.isEmpty() && !"unknown".equalsIgnoreCase(ip)) {
            return ip;
        }
        return request.getRemoteAddr();
    }
}
```

### WebSocket 代理

Spring Boot 集成 WebSocket 时，Nginx 需要额外配置 Upgrade 头：

```nginx
location /ws/ {
    proxy_pass http://127.0.0.1:8080;

    proxy_http_version 1.1;
    proxy_set_header Upgrade    $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host       $host;

    # WebSocket 长连接超时，默认 60s 会断开
    proxy_read_timeout 3600s;
    proxy_send_timeout 3600s;
}
```

### 代理超时配置

```nginx
location /api/ {
    proxy_pass http://127.0.0.1:8080;

    # 连接后端的超时
    proxy_connect_timeout 5s;
    # 读取后端响应的超时
    proxy_read_timeout 30s;
    # 向后端发送请求体的超时
    proxy_send_timeout 30s;
}
```

Java 后端有耗时接口（如报表导出、批量处理）时，`proxy_read_timeout` 需要调大，否则 Nginx 在 60s 后返回 504 Gateway Timeout。

## 负载均衡

### upstream 配置

[配图：Nginx 负载均衡架构图，展示客户端 → Nginx → 多个 Spring Boot 实例的请求分发]

```nginx
upstream backend {
    server 192.168.1.10:8080 weight=3;
    server 192.168.1.11:8080 weight=2;
    server 192.168.1.12:8080 weight=1;
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
    }
}
```

### 负载均衡策略

| 策略 | 指令 | 特点 | 适用场景 |
|------|------|------|---------|
| 轮询（默认） | 无需指定 | 按顺序逐个分配 | 实例性能一致 |
| 加权轮询 | `weight=N` | 按 weight 比例分配 | 实例性能不均 |
| IP Hash | `ip_hash` | 同一 IP 始终分配到同一后端 | 需要会话保持 |
| 最少连接 | `least_conn` | 分配给当前连接数最少的后端 | 请求处理时间差异大 |
| URL Hash | `hash $request_uri` | 同一 URL 始终分配到同一后端 | 缓存命中率优化 |

```nginx
# IP Hash — 会话保持
upstream backend_session {
    ip_hash;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}

# 最少连接 — 长耗时请求场景
upstream backend_long_poll {
    least_conn;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}

# URL Hash — 缓存优化
upstream backend_cache {
    hash $request_uri consistent;
    server 192.168.1.10:8080;
    server 192.168.1.11:8080;
}
```

### 健康检查

Nginx 开源版支持被动健康检查：

```nginx
upstream backend {
    server 192.168.1.10:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.11:8080 max_fails=3 fail_timeout=30s;
}
```

- `max_fails=3`：在 `fail_timeout` 时间内，失败 3 次后标记为不可用
- `fail_timeout=30s`：标记不可用后，30s 后再次尝试

Nginx Plus（商业版）支持主动健康检查，开源版可通过 `nginx_upstream_check_module` 第三方模块实现。

### Spring Boot 多实例部署示例

假设有 3 个 Spring Boot 实例，分别运行在 8080、8081、8082 端口：

```nginx
upstream springboot_cluster {
    least_conn;
    server 127.0.0.1:8080 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:8081 max_fails=3 fail_timeout=30s;
    server 127.0.0.1:8082 max_fails=3 fail_timeout=30s;

    # 备用节点，主节点全部不可用时才启用
    server 127.0.0.1:8083 backup;
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://springboot_cluster;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_connect_timeout 5s;
        proxy_read_timeout 30s;
    }
}
```

## 静态资源服务

### 前后端分离架构

[配图：前后端分离架构下 Nginx 同时服务静态资源和代理后端 API 的请求流向]

```nginx
server {
    listen 80;
    server_name www.example.com;

    # 前端静态资源
    location / {
        root /data/www/dist;
        index index.html;
        try_files $uri $uri/ /index.html;
    }

    # 后端 API 代理
    location /api/ {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
    }

    # 静态资源缓存
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2?)$ {
        root /data/www/dist;
        expires 30d;
        add_header Cache-Control "public, immutable";
    }
}
```

`try_files $uri $uri/ /index.html` 是 SPA 应用的关键配置。Vue/React 等单页应用的路由由前端处理，刷新页面时 Nginx 需要返回 `index.html` 而非 404。

### 文件上传代理

```nginx
location /api/upload {
    proxy_pass http://127.0.0.1:8080;

    # 上传文件大小限制
    client_max_body_size 50m;

    # 上传超时
    proxy_connect_timeout 300s;
    proxy_read_timeout 300s;
}
```

`client_max_body_size` 默认 1m，上传文件超过 1MB 会返回 413 Request Entity Too Large。

### Gzip 压缩

```nginx
http {
    gzip on;
    gzip_min_length 1k;
    gzip_comp_level 6;
    gzip_types
        text/plain
        text/css
        text/javascript
        application/javascript
        application/json
        application/xml
        image/svg+xml;
    gzip_vary on;
    gzip_buffers 16 8k;
}
```

| 参数 | 值 | 说明 |
|------|-----|------|
| `gzip_min_length` | 1k | 小于 1KB 不压缩，压缩后可能更大 |
| `gzip_comp_level` | 6 | 压缩级别 1-9，6 是 CPU 和压缩率的平衡点 |
| `gzip_types` | — | 默认只压缩 `text/html`，需显式添加其他类型 |

## HTTPS 配置

### SSL 证书配置

```nginx
server {
    listen 443 ssl;
    server_name api.example.com;

    ssl_certificate      /etc/nginx/ssl/example.com.pem;
    ssl_certificate_key  /etc/nginx/ssl/example.com.key;

    ssl_session_cache    shared:SSL:10m;
    ssl_session_timeout  10m;

    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers on;

    location / {
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto https;
    }
}

# HTTP 自动跳转 HTTPS
server {
    listen 80;
    server_name api.example.com;
    return 301 https://$host$request_uri;
}
```

### Spring Boot 感知 HTTPS

Nginx 做 SSL 终结后，Spring Boot 收到的是 HTTP 请求。如果后端需要生成 HTTPS 链接（如重定向、HATEOAS 链接），需配置：

```yaml
# application.yml
server:
  forward-headers-strategy: native
  tomcat:
    remoteip:
      protocol-header: X-Forwarded-Proto
      remote-ip-header: X-Forwarded-For
```

或使用 `ForwardedHeaderFilter`：

```java
@Bean
public ForwardedHeaderFilter forwardedHeaderFilter() {
    return new ForwardedHeaderFilter();
}
```

## 性能优化

### 连接与进程优化

```nginx
# 主配置 nginx.conf

# worker 进程数，auto = CPU 核心数
worker_processes auto;

# 每个 worker 的最大连接数
events {
    worker_connections 4096;
    # 允许一个 worker 同时接受所有新连接
    multi_accept on;
    # Linux 使用 epoll
    use epoll;
}
```

理论最大并发连接数 = `worker_processes × worker_connections`。4 核机器配置 `worker_connections 4096`，最大并发 16384。

### 内核参数调优

```bash
# /etc/sysctl.conf
# 允许 TIME_WAIT socket 复用
net.ipv4.tcp_tw_reuse = 1
# 减少 TIME_WAIT 超时
net.ipv4.tcp_fin_timeout = 15
# TCP 连接队列大小
net.core.somaxconn = 32768
# TCP 读缓冲区
net.ipv4.tcp_rmem = 4096 87380 16777216
# TCP 写缓冲区
net.ipv4.tcp_wmem = 4096 65536 16777216

# 生效
sudo sysctl -p
```

### 缓冲区优化

```nginx
http {
    # 请求体缓冲区，超过则写入临时文件
    client_body_buffer_size 16k;
    # 请求头缓冲区
    client_header_buffer_size 1k;
    # 大请求头缓冲区
    large_client_header_buffers 4 8k;

    # 代理响应缓冲
    proxy_buffering on;
    proxy_buffer_size 4k;
    proxy_buffers 8 16k;
    proxy_busy_buffers_size 32k;
}
```

### 静态资源缓存策略

```nginx
server {
    # HTML 文件：不缓存或短缓存（每次发版内容会变）
    location ~* \.html$ {
        root /data/www/dist;
        expires -1;
        add_header Cache-Control "no-cache, no-store, must-revalidate";
    }

    # JS/CSS：文件名带 hash，长缓存
    location ~* \.(js|css)$ {
        root /data/www/dist;
        expires 365d;
        add_header Cache-Control "public, immutable";
    }

    # 图片/字体：中等缓存
    location ~* \.(png|jpg|jpeg|gif|ico|svg|woff2?)$ {
        root /data/www/dist;
        expires 30d;
        add_header Cache-Control "public";
    }
}
```

前端构建工具（Vite/Webpack）输出的 JS/CSS 文件名带 content hash（如 `app.3a4b5c.js`），内容变化时文件名变化，可以安全设置长缓存。

## 流量控制

### 限流

```nginx
http {
    # 基于 IP 的请求速率限流，桶容量 10，速率 10 请求/秒
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

    server {
        location /api/ {
            # burst 允许突发 20 个请求排队，nodelay 不延迟直接处理
            limit_req zone=api_limit burst=20 nodelay;

            proxy_pass http://127.0.0.1:8080;
        }
    }
}
```

| 参数 | 值 | 说明 |
|------|-----|------|
| `rate=10r/s` | 10 请求/秒 | 每秒最多处理 10 个请求 |
| `burst=20` | 20 | 超出速率的请求排队，队列长度 20 |
| `nodelay` | — | 队列中的请求立即处理，不延迟 |
| `zone=api_limit:10m` | 10MB | 共享内存区域，存储 IP 状态，10MB 约记录 16 万个 IP |

### 并发连接数限制

```nginx
http {
    limit_conn_zone $binary_remote_addr zone=conn_limit:10m;

    server {
        location /api/ {
            # 每个 IP 最多 50 个并发连接
            limit_conn conn_limit 50;
            proxy_pass http://127.0.0.1:8080;
        }
    }
}
```

## 踩坑点 & 注意事项

### proxy_pass 末尾斜杠问题

```nginx
# 情况 1：proxy_pass 带 URI（含 /）
location /api/ {
    proxy_pass http://127.0.0.1:8080/;  # /api/user → /user
}

# 情况 2：proxy_pass 不带 URI
location /api/ {
    proxy_pass http://127.0.0.1:8080;   # /api/user → /api/user
}
```

`proxy_pass` 末尾有没有 `/`，行为完全不同。带 URI 时 Nginx 会替换匹配的 location 前缀；不带 URI 时原样转发。这是最常见的配置错误之一。

### root vs alias

```nginx
# root：拼接路径
# 请求 /static/app.js → 文件路径 /data/www/static/app.js
location /static/ {
    root /data/www;
}

# alias：替换路径
# 请求 /static/app.js → 文件路径 /data/assets/app.js
location /static/ {
    alias /data/assets/;
}
```

`root` 将 location 路径拼接到 root 后面，`alias` 将 location 路径替换为 alias 的值。`alias` 末尾必须加 `/`。

### 413 Request Entity Too Large

默认 `client_max_body_size` 为 1m。上传文件超过 1MB 返回 413 错误。需要在 `http`、`server` 或 `location` 块中调大：

```nginx
client_max_body_size 50m;
```

### 502/504 错误排查

| 错误码 | 含义 | 常见原因 |
|--------|------|---------|
| 502 Bad Gateway | Nginx 无法连接后端 | 后端进程未启动、端口错误、防火墙阻断 |
| 504 Gateway Timeout | 后端响应超时 | 后端处理慢、`proxy_read_timeout` 设置过小 |

排查步骤：

```bash
# 1. 检查后端进程是否存活
curl http://127.0.0.1:8080/actuator/health

# 2. 检查 Nginx 错误日志
tail -f /var/log/nginx/error.log

# 3. 检查端口监听
ss -tlnp | grep 8080

# 4. 检查防火墙
sudo iptables -L -n | grep 8080
```

### Host Header 丢失

不加 `proxy_set_header Host $host`，后端拿到的 Host 是 `upstream` 块中的地址（如 `127.0.0.1:8080`），导致：
- Spring Boot 的 `@RequestMapping` 基于 Host 的逻辑失效
- 重定向 URL 变成 `http://127.0.0.1:8080/...`
- Cookie domain 不匹配

### 跨域配置

Nginx 层处理 CORS 比在 Spring Boot 中处理更高效，避免请求到达后端：

```nginx
location /api/ {
    proxy_pass http://127.0.0.1:8080;

    # CORS
    add_header Access-Control-Allow-Origin  "https://www.example.com" always;
    add_header Access-Control-Allow-Methods "GET, POST, PUT, DELETE, OPTIONS" always;
    add_header Access-Control-Allow-Headers "Authorization, Content-Type" always;
    add_header Access-Control-Max-Age       3600 always;

    # OPTIONS 预检请求直接返回
    if ($request_method = OPTIONS) {
        return 204;
    }
}
```

**注意**：Nginx 层和 Spring Boot 层不要同时配置 CORS，否则会出现重复 Header 导致浏览器拒绝请求。

## 完整配置模板

以下是一个面向 Spring Boot 应用的生产级 Nginx 配置：

```nginx
worker_processes auto;

events {
    worker_connections 4096;
    multi_accept on;
    use epoll;
}

http {
    include       mime.types;
    default_type  application/octet-stream;

    # 日志格式
    log_format main '$remote_addr - $remote_user [$time_local] '
                    '"$request" $status $body_bytes_sent '
                    '"$http_referer" "$http_user_agent" '
                    '$upstream_response_time';

    access_log /var/log/nginx/access.log main;
    error_log  /var/log/nginx/error.log warn;

    # 性能优化
    sendfile    on;
    tcp_nopush  on;
    tcp_nodelay on;
    keepalive_timeout 65;

    # Gzip 压缩
    gzip on;
    gzip_min_length 1k;
    gzip_comp_level 6;
    gzip_types text/plain text/css text/javascript
               application/javascript application/json
               application/xml image/svg+xml;
    gzip_vary on;

    # 限流
    limit_req_zone $binary_remote_addr zone=api_limit:10m rate=10r/s;

    # 上游服务
    upstream springboot {
        least_conn;
        server 127.0.0.1:8080 max_fails=3 fail_timeout=30s;
        server 127.0.0.1:8081 max_fails=3 fail_timeout=30s;
    }

    # HTTP → HTTPS 重定向
    server {
        listen 80;
        server_name api.example.com;
        return 301 https://$host$request_uri;
    }

    # HTTPS 主服务
    server {
        listen 443 ssl;
        server_name api.example.com;

        ssl_certificate      /etc/nginx/ssl/example.com.pem;
        ssl_certificate_key  /etc/nginx/ssl/example.com.key;
        ssl_protocols        TLSv1.2 TLSv1.3;
        ssl_ciphers          ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
        ssl_prefer_server_ciphers on;
        ssl_session_cache    shared:SSL:10m;
        ssl_session_timeout  10m;

        client_max_body_size 50m;

        # API 代理
        location /api/ {
            limit_req zone=api_limit burst=20 nodelay;

            proxy_pass http://springboot;
            proxy_set_header Host              $host;
            proxy_set_header X-Real-IP         $remote_addr;
            proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
            proxy_set_header X-Forwarded-Proto https;

            proxy_connect_timeout 5s;
            proxy_read_timeout 30s;
            proxy_send_timeout 30s;
        }

        # WebSocket
        location /ws/ {
            proxy_pass http://springboot;
            proxy_http_version 1.1;
            proxy_set_header Upgrade    $http_upgrade;
            proxy_set_header Connection "upgrade";
            proxy_set_header Host       $host;
            proxy_read_timeout 3600s;
        }

        # 前端静态资源
        location / {
            root /data/www/dist;
            index index.html;
            try_files $uri $uri/ /index.html;
        }

        # 静态资源长缓存
        location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2?)$ {
            root /data/www/dist;
            expires 30d;
            add_header Cache-Control "public, immutable";
        }
    }
}
```

## 总结

Nginx 在 Java 后端架构中承担三个核心职责：流量入口（反向代理 + SSL 终结）、流量分发（负载均衡）、静态资源服务。配置时需重点关注：`proxy_set_header` 传递真实客户端信息、`proxy_pass` 末尾斜杠的替换行为、`client_max_body_size` 的上传限制、`try_files` 对 SPA 路由的支持。性能优化优先级：Gzip 压缩 > 静态资源缓存 > 连接参数调优 > 内核参数调优。生产环境务必配置 HTTPS、限流和健康检查。
